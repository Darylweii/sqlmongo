一句自然语言，系统自动走完整条数据理解与查询链路

1. 接口接入接收前端问题，整理本轮上下文和会话信息`message`、`history`、`session_id`、`alias_memory``effective_message`、`message_with_history`把“当前问题 + 历史上下文”组成一轮完整任务
2. QueryPlan 解析把自然语言翻译成标准查询任务单`user_query``query_plan`、`intent`、`query_mode`、`data_type`、`time_start`、`time_end`统一后续执行口径，避免直接拿口语去查数据库
3. 动作修正判断是走时序查询，还是直接走项目列表 / 统计等短路`query_plan`、`intent`、`resolved_time_range``override_action`、`override_reason`、`override_terminal`简单问题快速返回，避免所有请求都走重型链路
4. 作用域解析解析设备、项目、TG、对比对象，并结合会话记忆缩小范围`target`、`project_hints`、`alias_memory`、`comparison_targets``device_codes`、`device_names`、`tg_values`、`resolved_devices`把“a1_b9”“火锅店”“味多美”映射成系统认识的正式对象
5. 歧义澄清遇到多个同名设备时，返回候选卡片让用户确认`clarification_candidates``clarification_required`、候选设备列表、确认后的 alias memory保证结果准确，不默认乱选设备或跨项目汇总
6. 分片路由按时间范围、数据类型、标签确定要访问哪些集合`time_start`、`time_end`、`data_type`、`requested_tags``collections`、`data_tags`避免全库扫描，提升查询性能
7. 并行查询并发访问 MongoDB，拿回原始数据与基础统计`device_codes`、`collections`、`data_tags`、`time_start`、`time_end``raw_data`、`total_count`、`statistics`、`query_info`把“要查什么”真正落到数据库执行
8. 结果组织按回答风格生成结论、表格预览、智能分析和图表配置`statistics`、`query_plan`、`response_style`、`query_mode``final_response`、`show_table`、`table_preview`、`analysis`、`chart_specs`把原始数据变成业务和领导都能看懂的结果
9. SSE 返回把执行过程和最终答案流式推给前端所有中间节点结果`step_start`、`step_done`、`answer_delta`、`complete`让页面有过程感，也便于排查问题

语义层映射：系统是怎么“听懂人话”的

语义层不是替代数据库，而是先把用户说法映射成标准对象，帮助系统更稳地理解“项目名、设备名、指标名、俗称、同义词”。当前项目的语义资产主要由 `semantic_entries.json`、`semantic_layer.faiss`、`semantic_layer_metadata.json` 组成。

① 语义条目构建

从 MySQL 元数据和语义配置生成统一条目文件。

  • 来源脚本：`scripts/build_semantic_layer.py`
  • 输出文件：`data/semantic_entries.json`
  • 本质：把“项目 / 指标 / 设备”变成可检索文本

② 向量索引构建

把语义条目转成向量，写入 FAISS 索引。

  • 来源脚本：`scripts/build_vector_index.py`
  • 索引文件：`data/semantic_layer.faiss`
  • 元数据：`data/semantic_layer_metadata.json`

③ 查询时召回

用户提问后，语义层先召回相关项目、设备、指标候选。

  • 支持标准叫法
  • 支持简称、别名、关键词
  • 低置信度时进入候选确认

④ 映射到正式查询

最终仍会映射回结构化的 QueryPlan 和数据库条件。

  • 目标设备 `device_codes`
  • 目标标签 `data_tags`
  • 目标项目 / 作用域
🏢

Project 条目

用于理解项目名、项目简称和项目编码口径。

  • 字段示意：`project_id`、`project_name`、`project_code_name`
  • 作用：支持“北京电力项目”“智慧物联网能效平台”这类提问
📈

Metric 条目

用于理解指标名、标签名、单位、同义词。

  • 字段示意：`tag`、`name`、`unit`、`description`、`synonym`
  • 作用：把“用电量 / ep”“A 相电压 / ua”统一到标准指标
🔌

Device 条目

用于理解设备编码、设备名称、设备类型、所属项目和 TG。

  • 字段示意：`device_id`、`device_name`、`project_name`、`tg`
  • 作用：支持“a1_b9”“火锅店”“味多美”“电子楼总表”等说法
🧠

语义层在系统里的位置

语义层更多是“理解增强层”，不是最终执行层。它负责提高召回率和解释能力，但最终仍会回到结构化 QueryPlan 和正式数据库条件上执行。

🛡️

为什么不能只靠语义召回

因为真实业务里有重码设备、跨项目同名、显式设备码、时间和标签精细化条件，所以语义层只负责“找到候选”，不能越过正式的作用域确认和数据库执行规则。

系统最终返回给前端的内容

用户看到的不是一句孤立答复，而是一整组可读、可追踪、可核验的结果：答复文本、作用域说明、表格、智能分析、图表和执行步骤。

💬

最终答复

字段：`response` / `final_response`。根据 `response_style` 输出直接结论、对比说明或结构化分析。

📋

表格预览

字段：`show_table`、`table_type`、`table_preview`。用于展示原始明细、汇总表或项目列表。

📊

智能分析

字段：`analysis`、`focused_result`。展示均值、总量、趋势、波动、异常占比等结果。

📈

图表配置

字段：`chart_specs`、`show_charts`。供前端绘制趋势图、对比图和统计图。

`step_start`
告诉页面某个处理步骤已经开始，例如“解析 QueryPlan”“执行数据库查询”。
`step_done`
告诉页面某个步骤已完成，并附带耗时、摘要信息和部分查询上下文。
`answer_delta`
把最终回答按分段流式输出，用户不会一直空等。
`complete`
把最终完整结果一次性落地，包括答复、表格、分析卡片、图表和查询参数。

典型问题案例：更方便解释流程和语义层

下面把系统常见的 6 类问题都列出来：标准查询、对比、重码澄清、语义别名、快捷短路、异常分析。拿这几类去汇报，最容易把系统讲明白。

标准查询

“a2_b1 在 2024 年 1 月 1 日的 ua 是多少？”

这类问题信息最完整：设备码、时间、单相标签都很明确。

  • QueryPlan：sensor_query
  • data_type：ua
  • requested_tags：[ua]
  • 通常不需要语义层补充，只需做重码检查
对比查询

“a1_b9 与 b1_b14 哪个用电更多？”

系统会自动识别两个对比对象，生成 comparison 类型查询。

  • QueryPlan：comparison
  • targets：[a1_b9, b1_b14]
  • 默认指标：ep 用电量
  • 输出：对比结论 + 汇总表 + 智能分析
重码澄清

“查询 a1_b9 今天的电压数据”

如果多个项目里都有 a1_b9，系统不会直接查，而是先要求确认具体是哪一个设备。

  • 先命中多个候选设备
  • 返回候选卡片供用户选择
  • 确认后写入会话作用域记忆
  • 再继续原问题
语义别名

“味多美和火锅店的用电量对比”

这类问题通常没有显式设备码，需要语义层先把“商户名/俗称”映射成候选设备。

  • 语义层先召回“味多美”“火锅店”相关设备
  • 若候选唯一，可直接进入 QueryPlan 执行
  • 若候选不唯一，再进入澄清分支
  • 这类问题最能体现语义层价值
快捷短路

“有哪些项目可用？”

这是典型目录型问题，不需要走 MongoDB 时序查询链路。

  • Action Override 直接改成 list_projects
  • 跳过 Metadata / Router / Fetch
  • 直接返回项目列表表格
异常分析

“a1_b9 设备 2024 年 1 月有没有异常用电时间点？”

这是分析型问题，不仅要查数据，还要做异常点识别与结果组织。

  • QueryPlan：anomaly_points
  • 通常会查 ep 或相关能耗标签
  • 输出：异常点数量、时间点、占比、摘要说明
用户原话系统理解重点语义层是否参与是否可能澄清最终落到什么执行
a2_b1 在 2024-01-01 的 ua 是多少明确设备码 + 明确时间 + 明确单相标签低，主要做重码检查若 a2_b1 多项目重码，则需要单标签时序查询：ua
a1_b9 与 b1_b14 哪个用电更多识别为对比查询中，帮助补足默认指标 / 语义理解任一设备重码时需要对比查询：comparison + ep
味多美和火锅店的用电量对比先从商户名映射设备候选高，是这类问题的关键若候选不唯一则需要语义召回 → 设备确认 → 对比查询
有哪些项目可用识别为目录型问题低通常不需要快捷短路：list_projects
a1_b9 今天的电压数据设备码明确，但数据类型较宽中若 a1_b9 多项目重码则需要时序查询：电压相关标签
北京电力项目有哪些设备先理解项目范围，再列设备高，项目名口径需映射项目命中不清晰时可能需要设备列表 / 项目范围列表
