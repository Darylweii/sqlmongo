# 查询路由分层说明

本文档定义 `sqlmongo2` 项目当前的查询路由分层策略，用于区分：
- 哪些问题应直接走快路径
- 哪些问题必须进入实体解析层
- 哪些问题必须保留给 LLM 做理解、推理和表达

目标不是“把所有问题都写规则”，而是把 **高确定性参数抽取** 与 **开放式语义理解** 分开，降低首轮延迟并提升稳定性。

## 总体原则

1. **P0 快路径** 只处理高确定性、低歧义、可直接结构化抽取的问题。
2. **P1 实体解析** 处理“用户说法不标准，但仍可通过候选召回解决”的问题。
3. **P2 LLM** 处理开放式理解、复杂推理、总结生成、多轮上下文等问题。
4. **不要把业务私有知识写死到代码里**，尤其是商户名、别名、俗称与设备号映射。

## 路由优先级

当前推荐的执行顺序如下：

1. 提取当前问题文本
2. 识别显式设备号 / 会话已确认设备
3. 解析时间范围、指标类型、分页、过滤条件
4. 命中固定意图时直接走快路径
5. 未命中快路径时进入实体解析层
6. 仍有歧义时触发候选确认
7. 需要推理、解释、总结时进入 LLM

## P0：可快路径化问题清单

### 适用条件

满足以下至少大部分条件时，优先走 P0：
- 出现显式设备号，如 `a1_b9`、`b1_b14`
- 时间表达明确，如 `今天`、`昨天`、`本周`、`本月`、`2024年1月`
- 指标明确，如 `电压`、`电流`、`功率`、`用电量`
- 问题目标固定，如“查数据”“列项目”“查数据类型”
- 不需要歧义消解，不依赖额外业务知识

### 典型问题

- `查询 a1_b9 设备今天的电压数据`
- `a1_b9 设备2024年1月的用电量是上升还是下降？`
- `比较 a1_b9、b1_b14、a2_b3 最近一周的用电情况`
- `a1_b9 有哪些数据类型？`
- `有哪些项目可用？`
- `哪个项目设备最多？`
- `查询 a1_b9 最近一周功率大于100的数据`
- `查询 a1_b9 今天的电流数据，每页50条`

### 当前代码映射

- 时间解析：`src/agent/orchestrator.py:386`
- 显式设备号提取：`src/agent/orchestrator.py:1109`
- 数据类型意图识别：`src/agent/orchestrator.py:1239`
- 快路径构造：`src/agent/orchestrator.py:1255`
- 首轮优先快路径：`src/agent/orchestrator.py:500`、`src/agent/orchestrator.py:666`

### P0 应负责的内容

- 设备号抽取
- 时间范围换算
- 指标类型推断
- 项目列表 / 项目统计固定意图
- 数据类型探测固定意图
- 分页参数、过滤参数这类明确结构化字段

### P0 不应负责的内容

- 商户别名映射
- 项目私有叫法映射
- 多轮开放式语义推理
- 原因解释、趋势归因、自然语言总结

## P1：必须走实体解析的问题清单

### 适用条件

问题中没有稳定的显式设备号，但仍可通过“词法匹配 + 向量召回 + 候选澄清”解决。

### 典型问题

- `味多美对比火锅店的用电量`
- `电子楼那个总表最近一周的电压情况`
- `1号楼配电室的主表本月怎么样`
- `北京电力项目里味多美的总表`
- `a1b9 最近一周用电情况`（设备号变体）
- `A1-B9 今天电流`（设备号格式变体）
- `北区空调总表昨天的数据`
- `对比 a9_b6 a1_b7、a2_b14 和 a1_b9 最近一周的用电情况`

### 当前代码映射

- Chroma 实体解析器：`src/entity_resolver/chroma_resolver.py:29`
- 设备目录拉取：`src/metadata/metadata_engine.py:356`
- 主设备候选召回入口：`src/agent/orchestrator.py:1505`
- 候选筛选：`src/agent/orchestrator.py:1531`
- 是否触发澄清：`src/agent/orchestrator.py:1552`

### P1 应负责的内容

- 设备别名、商户名、位置名等非标准说法召回
- 词法匹配与向量检索融合排序
- 候选列表输出
- 歧义时追问或一键确认
- 同会话确认后的别名学习

### P1 不应负责的内容

- 开放式分析解释
- 业务原因推断
- 长文本总结生成
- 图表说明文案生成

## P2：必须走 LLM 的问题清单

### 适用条件

问题不是简单参数抽取，而是需要理解、推理、总结、归因或多轮上下文关联。

### 典型问题

- `a1_b9 设备2024年1月的用电量是上升还是下降？为什么？`
- `这周的波动为什么比上周大？`
- `帮我分析一下最近有没有异常`
- `比较这三个设备的变化趋势，并给出重点结论`
- `哪个设备最稳定，原因可能是什么？`
- `把结果总结成一份运营分析`
- `根据这些数据给我一些节能建议`
- `刚才那几个设备里最稳定的是谁？那它昨天怎样？`

### 当前代码映射

- LLM 决策：`src/agent/orchestrator.py:933`
- JSON 响应解析：`src/agent/orchestrator.py:968`
- 结果观察提示：`src/agent/orchestrator.py:256`
- 结构化分析文案生成：`src/agent/orchestrator.py:1939`

### P2 应负责的内容

- 模糊问题理解
- 开放式语义推理
- 多轮上下文承接
- 结果总结与表达
- 趋势解读、异常说明、后续建议

## 明确禁止写死为规则的内容

以下内容 **绝不应直接写成 P0 规则**：

- 商户名到设备号的固定映射，如“味多美 = a1_b5”
- 某个项目里的内部别名词典硬编码
- 针对单一失败案例的 `if query == ...`
- 把自然语言表达方式逐句穷举为规则
- 把开放式业务知识塞进参数抽取逻辑

正确做法是：
- 别名问题走 P1 实体解析
- 解释与归因问题走 P2 LLM
- 用户确认后的结果只存会话级缓存，不写死为全局常量

## 当前建议的决策表

| 问题类型 | 推荐层级 | 原因 |
| --- | --- | --- |
| 显式设备号 + 明确时间 + 明确指标 | P0 | 可稳定结构化抽取 |
| 显式设备号 + 数据类型探测 | P0 | 固定意图，无需推理 |
| 项目列表 / 项目统计 | P0 | 固定意图，无需语义理解 |
| 商户名 / 俗称 / 位置描述找设备 | P1 | 需要召回候选但不一定需要 LLM |
| 多候选设备需用户确认 | P1 | 应先澄清，不应硬猜 |
| 开放式分析、归因、建议 | P2 | 需要推理和生成 |
| 多轮上下文承接 | P2 | 需要结合历史理解 |

## 落地建议

1. 持续扩大 P0，但只扩“参数抽取型能力”，不要扩业务知识硬编码。
2. 把非标准设备表达全部压到 P1 实体解析层处理。
3. 让 P2 专注在高价值问题：解释、分析、总结、推理。
4. 每新增一种快路径，都补一个真实 smoke case，验证：
   - 是否减少首轮耗时
   - 是否没有误伤 P1 / P2 问题
   - 是否仍保持现有 API 输出兼容
